# 피그마 → 퍼블리싱 코드 생성 (웹/기본 버전)

> 스킬 버전: 1.5.0 — 이 줄의 버전 번호가 최신인지 확인하세요. 다른 대화창/세션에서 작업한 버전과 다르면 아직 동기화가 안 된 것입니다.

**이 스킬은 웹(기본) 버전을 만드는 것까지만 담당한다. 태블릿/모바일 반응형은 `pub-responsive` 절차로 넘긴다.** 작업 시작 시 1-4에서 일반/반응형을 먼저 명확히 구분하고, 반응형이면 사용자가 별도 명령을 입력할 필요 없이 이 흐름 그대로 `pub-responsive`의 절차로 자동 이어간다.

## 사용자에게 질문할 때 (전체 공통)

이 스킬에서 나오는 모든 선택형 질문(영역 선택, 새/기존 페이지 선택, 파일 선택 등)은 **반드시 `AskUserQuestion` 도구를 사용**한다. 그냥 텍스트로 나열만 하고 타이핑으로 답하게 하지 않는다.
- **질문 문구와 선택지 라벨은 한글로 작성한다.** `main.html`, `sub.html` 같은 실제 파일명 자체는 영문 그대로 두되, 그 옆에 한글 설명을 병기한다.
- `AskUserQuestion`의 "Type something." / "Chat about this" 같은 고정 UI 문구는 도구 자체의 고정 표시라 스킬로는 바꿀 수 없다.

## 작업 전제

- 이 스킬은 반드시 실제 프로젝트 폴더 안에서 실행 중인 Claude Code 세션에서 사용한다.
- **작업 폴더 안에 영역이 명확할 때만 그 폴더가 곧 "영역"이다.** 상위 폴더에 web1/web2/bcnimda처럼 여러 영역이 함께 있으면 자동으로 정하지 말고 반드시 사용자에게 먼저 물어본다.
- layout.css, design.css는 실제 파일이다. git으로 커밋되면 다른 퍼블리셔와 자연스럽게 공유된다.

## 작업 절차

1. **아래 순서를 매번 무조건, 하나씩 순서대로 진행한다.**

   **1-1. 영역 확인**
   - 항상 지금 Claude Code 세션이 실행된 현재 폴더를 기준으로 진행한다. "실제 프로젝트 폴더 경로를 알려달라"거나 다른 폴더로 옮겨서 작업하자는 식으로 되묻지 않는다.
   - 현재 폴더 또는 `inc/` 아래에 이미 만들어진 영역 폴더가 있으면, 그 이름을 선택지로 제시하고 "기타(직접 입력)"도 함께 제시한다.
   - 영역 폴더가 하나도 없어도 마찬가지로 현재 폴더에서 바로 물어본다.
   - 지금 실행 위치가 이미 특정 영역 폴더 안이라 다른 후보가 아예 없을 때만 예외적으로 이 단계를 묻지 않는다.

   **1-2. 새 페이지인지 기존 페이지인지 확인**
   - `html/{영역}/` 폴더 안의 파일 목록을 실제로 조회한다. 폴더가 없거나 비어 있으면 새 페이지로 진행. 파일이 하나 이상 있으면 반드시 물어본다: "새 페이지를 만들까요, 아니면 기존 페이지에 이어서 작업할까요?"

   **1-3. (기존 페이지인 경우) 대상 파일 선택**
   - 파일 목록을 `AskUserQuestion`으로 제시해서 화살표 키로 고를 수 있게 한다. 10개 이상이면 이름 검색으로 먼저 좁힌다.

   **1-4. 일반/반응형 구분 + 피그마 프레임 지정 확인 + (새 페이지인 경우) 파일명 자동 결정**
   - **가장 먼저 `AskUserQuestion`으로 명확히 구분해서 물어본다**: "이번 작업은 일반(웹 버전만)인가요, 반응형(웹+태블릿+모바일)인가요?" — 사용자의 원래 요청에 "반응형", "태블릿", "모바일" 같은 말이 이미 있었다면 그 답이 명확하니 다시 묻지 않고 바로 그쪽으로 진행한다.
     - **일반이면**: 아래 순서대로 이 스킬(`/pub`) 안에서 계속 진행한다. 웹 피그마 링크를 물어본다('Copy link to selection'으로 복사한 node-id 포함 링크).
     - **반응형이면**: 사용자에게 별도 명령을 다시 입력하라고 시키지 않는다. **이 흐름 그대로, `pub-responsive` 스킬(`/mnt/skills/.../pub-responsive/SKILL.md` 또는 해당 플러그인 경로)의 절차를 이어서 그대로 따른다** — 사용자 입장에서는 `/pub` 한 번 실행한 것으로 끝까지 이어지는 것처럼 보여야 한다. 다만 웹 버전 생성(1~6번, 고정 규칙)까지는 이 스킬대로 하고, 그 다음 반응형 단계(웹→태블릿→모바일 순차, 완전 대조, 커버리지 표 등)부터는 `pub-responsive`에 정의된 절차와 형식을 그대로 적용한다. 두 스킬의 규칙을 다르게 섞어 쓰지 않는다 — 반응형 부분은 전적으로 `pub-responsive`의 규칙을 따른다.
   - **같은 프레임을 다시 요청받으면 기본적으로 "재검증 후 수정" 모드로 동작한다.** 기존 구현이 있으면 자기 점검으로 재검증 후 문제 있는 부분만 고친다. "처음부터 다시 만들어줘"일 때만 완전 재생성.
   - **새 페이지인 경우, 파일명을 따로 물어보지 않고 프레임 이름 기반으로 자동 결정한다.**
     - 메인/대표 페이지는 `index.html`이 아니라 `main.html`.
     - `sub.html`, `list.html`, `detail.html`처럼 구분 안 되는 이름 금지 — 프레임 이름 반영해서 구체적으로.
     - 겹치면 더 구체화. 자동 결정 후 "이 이름으로 만들었어요, 마음에 안 들면 말씀해주세요" 정도로만 안내.

   **공통 규칙**
   - **한 번 확인된 항목은 다시 묻지 않는다.** 단, 이 범위는 지금 실행 중인 `/pub` 한 번의 호출 안에서만이다 — `/pub`을 다시 실행하면 매번 새 작업으로 취급하고 1-1부터 다시 시작한다.
   - 확인 완료되면 `inc/{영역}/css/`, `html/{영역}/`, `files/{영역}/images/common/` 구조가 없을 경우 생성한다. 별도로 안 물어봐도 된다.
   - **작업 도중 영역 전환** 요청이 오면 즉시 전환하고 1-2부터 다시 진행, 이전 영역과 안 섞이게 한다.
2. 확인된 영역 기준 `layout.css`, `design.css`를 찾는다.
   - **토큰 최적화**: design.css가 크면 관련 키워드로 grep해서 관련 블록만 확인, 전체를 매번 읽지 않는다(300줄 이하이거나 첫 실행/대개편일 땐 전체 확인).
   - 폴더/파일이 없으면 새로 만든다.
3. 연결된 Figma 도구로 `$ARGUMENTS`로 받은 링크의 프레임을 분석한다.
   - **공통 레이아웃(GNB/LNB/헤더/푸터) — 서브 페이지에만 적용**: GNB/LNB/헤더/푸터가 포함돼 있으면 다시 분석해서 새로 만들지 않는다. 메인 페이지(`main.html`)는 이 규칙 예외 — 항상 피그마 메인 프레임 기준으로 그대로 만든다. 재사용 범위는 `<header>`/`<nav>`/`<footer>` 태그나 `gnb`/`lnb`/`header`/`footer`/`nav` 클래스로 식별되는 요소로 엄격히 한정하고, 다른 콘텐츠 섹션을 같이 끌어오지 않는다. 기존 서브 페이지에서 grep으로 그 부분만 찾아 복사, 없으면 이번에 만들고 "기준 버전"이라고 안내. 디자인이 달라 보이면 임의로 새로 만들지 말고 확인.
   - **노드 완전성 확인(필수)**: 스크린샷엔 보이는데 참고 코드에 없는 시각 속성(테두리/배경/그림자)이 있으면 부모 노드 누락을 의심하고 부모까지 포함해서 다시 조회한다.
   - **속성 1:1 보존(필수)**: border, background, border-radius, box-shadow, 정확한 색상/변수, padding/gap 수치를 하나도 빠짐없이 그대로 옮긴다. 프레임워크 문법만 걷어내고 구조/수치는 유지.
   - **부모-자식 그룹핑 구조 보존(필수)**: 참고 코드에서 형제로 같이 있는 요소는 구현에서도 반드시 같은 부모 안에 형제로 둔다. 값은 맞아도 그룹 구조가 달라지면 레이아웃이 어긋난다.
   - **정확한 값은 추측하지 않고 변수 정의를 조회**: 애매한 색상/이펙트는 `Figma:get_variable_defs`로 확인, 눈대중 금지.
   - **토큰 최적화**: Figma 도구는 항상 특정 프레임/노드 하나만 지정해서 조회, 페이지/파일 전체 조회 금지.
   - **차트/그래프류는 예외 — 먼저 물어본다**: 방사형/도넛/막대/선 그래프는 SVG 인라인을 자동 적용하지 않고 "SVG로 재현할지 이미지로 대체할지" 먼저 확인한다. 이미지로 결정되면 막대/선/격자선을 쪼개지 않고 통 이미지 하나로 처리(범례/제목 등 차트 밖 요소는 기존 규칙대로 마크업).
4. 아래 [고정 규칙]에 따라 HTML을 생성한다.
   - **CSS 작성 전에 치수표를 먼저 만든다(필수, 추가 API 호출 없이 3번에서 받은 데이터로).**
     ```
     | 요소 | 너비 | 높이 | 비고 |
     |---|---|---|---|
     | 프로필 카드 | 475px | 416px | 고정 |
     ```
   - 여러 컬럼이 나란히 배치된 레이아웃이면, 컬럼별 (높이 합 + gap 합)이 서로 맞는지 CSS 작성 전에 미리 계산한다.
   - 요소가 한두 개뿐인 단순한 프레임이면 표를 간단히 줄여도 된다.
5. **1-3(기존 페이지) 또는 1-4(새 페이지)에서 확인/결정한 파일에 저장한다.** (다시 안 물어봄) `html/{영역}/`에 저장, `layout.css`/`design.css` 실제 갱신, 이미지는 `files/{영역}/images/common/`. **JS가 필요하다고 요청받았으면 `inc/{영역}/js/`에 만든다** (공용 라이브러리는 기존 `inc/lab/js/` 그대로).
6. 자기 점검 후 결과를 보고한다.

## layout.css가 비어있는 첫 실행일 때

피그마에서 다음을 먼저 뽑아 `:root` 토큰으로 확정한다: 가장 많이 쓰이는 폰트 패밀리(`--font-family-base`), 자주 쓰이는 폰트 크기들(`--font-size-*`, 빈도 높은 걸 base로), 가장 흔한 line-height(`--line-height-base`), 자주 쓰이는 색상(`--color-*`). 이후 모든 컴포넌트는 이 변수만 참조한다.

회사 표준 브레이크포인트(태블릿 `max-width: 1295px`, 모바일 `max-width: 744px`)도 이 시점에 layout.css 상단에 주석으로 문서화해둔다: `/* 브레이크포인트: 태블릿 max-width 1295px, 모바일 max-width 744px (회사 표준, 프로젝트 전체 공통) */` — 실제 브레이크포인트 CSS 작성은 `pub-responsive`가 담당하지만, 토큰 문서화는 여기서 한다.

## 고정 규칙 (절대 어기지 않음)

1. **이미지/아이콘**:
   - **SVG(벡터)**: 실제 이미지 파일로 만들지 않고 `<svg>...</svg>`를 HTML 안에 인라인으로 삽입한다. `fill="currentColor"`로 색상을 CSS에서 제어할 수 있게 한다. 이건 규칙 7(인라인 스타일 금지) 대상이 아니다.
   - **반복 SVG의 `id` 충돌 방지(필수)**: `<defs>`의 `linearGradient`/`mask`/`clipPath`/`filter` id가 같은 페이지에서 여러 번 반복되면 두 번째부터 깨진다. 인스턴스마다 id를 고유하게(`-1`, `-2` 등) 만든다.
   - **래스터 이미지**: 추출 가능하면 `files/{영역}/images/common/파일명`으로 저장. 불가능하면 임의로 만들지 않고 크기만 잡은 빈 영역(전용 클래스로 크기 정의)으로 처리.
   - SVG인지 래스터인지 애매하면 임의로 정하지 말고 물어본다.
   - **크롭 정확도**: 부모 프레임/여백까지 통째로 잘라내지 않고, 실제 이미지 경계와 일치하는 가장 안쪽 노드만 추출한다.
   - **크롭 검증 및 재크롭**: 다운로드 파일의 실제 픽셀 크기와 피그마 레이아웃 표시 크기를 비교, 다르면 ImageMagick/Pillow 등으로 재크롭. 좌표 판단이 애매하면 전/후 비교 이미지로 확인받는다.
   - **크기 보정**: `<img>`에 `object-fit: cover; object-position: center;` 기본 적용.
   - **확장자-내용 일치 검증(필수)**: 저장 시 `file` 명령어 등으로 확장자와 실제 포맷이 맞는지 확인, 불일치 시 바로잡는다.
2. **폼/인터랙션 요소는 실제 시맨틱 HTML로 만든다**: 드롭다운은 `<select>`, 체크박스/라디오는 `<input type="checkbox/radio">`, 토글은 `<input type="checkbox" role="switch">` + `checked`/`aria-checked`. div/span으로 흉내만 내는 것 금지.
   - **접근성**: `accessibility-prompt-rules` 스킬이 있으면 WCAG 2.1 AA 기준도 같이 적용.
   - **JS 없이 작동 안 하는 요소**(탭/아코디언/모달)는 발견 시 조용히 넘어가지 않고 "마크업만 있고 JS 필요, 만들어드릴까요?"를 보고에 명시.
3. **hover/focus**: 피그마에 실제로 표현된 상태만 구현, 추측 금지.
4. **클래스 재사용**: 기존 design.css 먼저 확인, 비슷한 게 있으면 재사용, 완전히 새로울 때만 신규 클래스. 애매하면 규칙 12.
5. **클래스 네이밍**: kebab-case + BEM, 블록명 앞에 컴포넌트/페이지 접두사 필수(`.tag`, `.input` 같은 일반명 금지). 기존 무접두사 클래스 전체 리네이밍은 임의로 안 하고 규칙 12에 따라 확인. **아이콘 버튼**은 매번 `display:flex;align-items:center;justify-content:center;` + svg `width`/`height` 명시.
6. **색상/폰트/사이즈**: 하드코딩 금지, `var(--...)`만 참조, 없으면 layout.css에 추가.
7. **스타일 작성 위치**: `layout.css`/`design.css`에만 작성, HTML에 `<style>`/`style=""` 절대 금지.
8. **리셋 범위 검증**: `<p>`,`h1~h6`,`ul/ol/li`,`button`,`fieldset/legend`,`input/select/textarea`,`figure` 등 브라우저 기본 여백/테두리가 layout.css 리셋에 커버되는지 확인, 없으면 추가.
   - **`box-sizing: border-box` 전역 적용(필수)**: `*, *::before, *::after { box-sizing: border-box; }`가 layout.css에 있는지 확인, 없으면 추가.
9. **웹표준 준수**: `zoom` 등 비표준/레거시 속성, 벤더 핵 금지. Flexbox/Grid/`%`/`aspect-ratio`/`clamp()` 등 표준 CSS로 구현.
10. **긴 텍스트/실제 데이터 대비**: 고정 너비/줄수 텍스트 영역엔 오버플로우 처리 기본 적용 — 한 줄은 `overflow:hidden;text-overflow:ellipsis;white-space:nowrap;`, 여러 줄은 `-webkit-line-clamp:N` 방식. 배지/태그처럼 짧고 고정된 값은 생략 가능, 애매하면 규칙 12.
11. **요청 범위를 벗어난 콘텐츠 추가 금지**: 결과물의 모든 섹션은 (a) 이번 프레임 내용이거나 (b) 정당하게 재사용한 GNB/LNB/헤더/푸터, 둘 중 하나로 설명돼야 한다.
12. **애매한 판단**: 재사용 여부, 정보 부족, 그 밖의 임의 결정 상황은 반드시 먼저 사용자에게 물어본다.

## 생성 후 자기 점검 (사용자에게 보고하기 전에)

같은 세션에서 [자기 점검 체크리스트]를 빠르게 확인하는 것으로 평소엔 충분하다. `figma-review-agent`(Opus, 독립 컨텍스트)는 아래 "픽셀 단위 검증 루프"에서 정한 경우에만 호출한다.

- `figma-review-agent`는 플러그인에 이미 포함되어 배포된다(`pub/agents/figma-review-agent.md`). 매번 있는지 확인하거나 새로 만들려 하지 않는다 — 필요할 때 Task 도구로 이름만 호출.

## 픽셀 단위 검증 루프 (기본은 가볍게, 필요할 때만 정밀하게)

기본은 매 섹션마다 안 돌린다. 사용자의 명시적 요청 / 페이지 전체 최종 확정 시점 / 이미 차이를 발견해 정확한 위치가 필요할 때만 돌린다.

**토큰 절약**: 피그마 재조회 안 함(생성 세션의 스크린샷/데이터 재사용), Opus는 기본 미사용(반복 실패 시에만), 반복 최대 1~2회, 기준치 3%.

`pub/scripts/capture.js`/`diff.js`가 있으면: 캡처 → 피그마 원본과 비교(기준치 3%) → FAIL이면 diff 이미지 좌표로 구체 수정 후 최대 1~2회 재시도 → 그래도 안 되면 남은 차이를 사용자에게 보고. 스크립트가 없으면 체크리스트만으로 진행하되 그 사실을 알린다.

## 자기 점검 체크리스트

그룹별로 한 번에 훑는다.

1. **시각적 비교**: 생성 결과와 (3번에서 이미 받은, 재조회 안 하는) 스크린샷을 비교, 다른 부분은 위치/정도로 구체적으로 짚는다. 없어진 시각 속성 있으면 부모 노드 누락 의심. 100% 재현 어려운 부분은 근사치임을 밝힌다. 피그마에서 같은 줄에 있던 요소가 결과물에서 분리돼 있으면 그룹핑 구조를 다시 대조해서 고친다.
2. **시맨틱/기능 + 코드 위생**: select/checkbox/radio/toggle 시맨틱 여부, 인라인 style/`<style>` 없음, `zoom` 등 비표준 없음, 이미지 확장자-내용 일치, 반복 SVG id 고유성을 한 번에 훑는다.
3. **레이아웃/범위 무결성**: 다컬럼 카드 높이 합 일치(피그마 고정 높이 반영), `box-sizing: border-box` 리셋 유무로 인한 페이지 전체 가로 스크롤 없음, 모든 섹션이 (a) 이번 프레임 (b) 정당한 GNB/LNB/헤더/푸터 재사용 중 하나로 설명되는지(규칙 11) 확인.

문제 발견 시 보고 전에 고친다. 애매하면 규칙 12.
