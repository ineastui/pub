> 스킬 버전: 2.5.0

# 웹 페이지에 태블릿/모바일 반응형 추가

이 스킬은 **딱 하나의 일만** 한다: 이미 있는(또는 지금 만드는) 웹(데스크톱) CSS에, 태블릿과 모바일 오버라이드를 빠짐없이 추가하는 것. 그 외의 것(폼 요소 시맨틱 처리, 이미지 규칙, GNB/LNB 재사용, 클래스 네이밍 등)은 `/pub` 스킬의 몫이니 여기서 다시 다루지 않는다.

**이미 만든 태블릿/모바일 결과를 수정하려는 요청이면(새로 만드는 게 아니라)**, 아래 순서를 처음부터 다시 밟지 않는다 — [수정 요청 처리] 섹션으로 바로 간다.

**링크는 한 번에 여러 개를 받지 않는다. 웹 → 태블릿 → 모바일 순서로 하나씩 받아서, 그 단계를 완전히 끝내고 나서 다음 링크를 요청한다.** 한꺼번에 여러 링크를 받으면 뒤쪽(특히 모바일)을 놓치는 경우가 실제로 반복됐다 — 단계를 쪼개서 한 번에 하나만 처리하는 게 훨씬 확실하다.

**"링크를 받는다"는 반드시 사용자가 직접 타이핑하거나 붙여넣은 링크만을 뜻한다.** 피그마 파일을 검색하거나 목록을 조회해서 "이게 태블릿/모바일 프레임인 것 같다"고 스스로 찾아서 쓰지 않는다 — 이름이 같은 프레임이 여러 개 있는 경우가 많아서 자동으로 찾으면 잘못된 프레임을 쓸 위험이 크다. 사용자가 링크를 주기 전까지는 어떤 노드도 조회하지 않는다. 사용자가 링크 없이 "있다"/"없다"로만 답하면, 있다고 했는데 링크를 안 줬다면 다시 링크를 요청한다 — 알아서 찾지 않는다.

**받은 링크가 정말 "같은 컴포넌트의 다른 크기"가 맞는지 확인한다(중요).** 사용자가 준 두 링크가 서로 완전히 다른 제목/내용을 가진, 그냥 나란히 배치될 별개의 컴포넌트(예: "나의 상담 현황" 카드와 "나의 디지털배지 취득 현황" 카드처럼 각각 독립적인 콘텐츠)일 수도 있다 — 이런 경우는 반응형 관계가 아니다. 프레임을 조회한 뒤 제목/구조가 서로 다른 컴포넌트로 보이면(단순히 크기나 배치만 다른 게 아니라 아예 다른 내용이면), 계속 진행하기 전에 사용자에게 확인한다: "이 두 프레임이 같은 컴포넌트의 웹/태블릿 버전이 맞나요, 아니면 서로 다른 콘텐츠인가요?"

## 1단계 — 웹

1. **대상 CSS 파일 경로를 확인한다** (예: `inc/{영역}/css/design.css`).
2. **웹 피그마 링크를 하나 받는다.** 아직 없으면 이것부터 물어보되, 처음 요청할 때 전체 흐름을 먼저 짧게 안내한다: "반응형은 웹 → 태블릿 → 모바일 순서로 하나씩 진행할게요. 먼저 웹 프레임 링크를 알려주세요." 이렇게 처음부터 전체 순서를 알려줘야, 왜 지금 링크 하나만 요청하는지 사용자가 헷갈리지 않는다(이 스킬을 처음 쓰는 사람도 있으므로).
3. 이 링크로 가리키는 컴포넌트가 대상 CSS 파일에 이미 만들어져 있는지 확인한다. 없으면(웹 버전 자체가 아직 없으면) `/pub`으로 웹 버전부터 만들어달라고 안내하고 여기서 중단한다.
4. 웹 프레임을 Figma 도구로 조회한다(해당 노드만, 페이지 전체 조회 금지). 참고 코드를 최상위부터 리프 요소까지 훑어서, 이번에 반응형을 적용할 요소들의 목록을 정리한다(이게 다음 단계 대조의 기준이 된다).
5. **이 단계가 끝났다고 사용자에게 알리고, 태블릿 링크를 요청한다.** 웹 단계 결과를 먼저 간단히 보여준다(예: "웹 프레임 분석 완료 — .card, .card__img, .card__body 등 N개 요소 확인. 이제 태블릿 링크를 알려주세요.")

## 2단계 — 태블릿

1. **태블릿 피그마 링크가 있는지 먼저 확인한다.** `AskUserQuestion`으로 묻는다: "태블릿 전용 피그마 프레임이 있나요, 아니면 없어서 웹 기준으로 안 깨지지 않게만 처리할까요?"
   - **태블릿 프레임이 있으면**: 그 링크를 받아서 아래 2~4번대로 정확히 대조해서 만든다.
   - **태블릿 프레임이 없으면**: 구체적인 태블릿 전용 디자인을 추측해서 만들지 않는다. 대신 웹 레이아웃을 기준으로 "화면이 안 깨지고 자연스럽게 흐르는" 수준의 합리적인 처리만 한다(예: 고정폭 요소를 `width:100%`/`flex-wrap`으로 유동화, 지나치게 넓은 gap/padding을 살짝 줄이는 정도). 세부적인 재배치(요소 순서 변경, 새로운 레이아웃 구조)는 하지 않는다 — 애매하면 규칙에 따라 사용자에게 확인한다. 이 경우 4번(보고)에 "태블릿 전용 프레임 없이 웹 기준으로 유동 처리함"이라고 명시한다.
2. (태블릿 프레임이 있는 경우) 태블릿 프레임을 조회한다(해당 노드만). 1단계에서 정리한 요소 목록을 기준으로, **하나하나 웹과 태블릿을 대조**한다 — 부모 컨테이너만 보고 끝내지 않는다. 부모의 배치(행↔열)가 바뀌었다고 자식(카드 안 텍스트, 아이콘, 배지 등)도 자동으로 확인됐다고 넘겨짚지 않는다.
3. 각 요소마다 `display`, `flex-direction`, `align-items`, `justify-content`, `gap`, `width`, `height`, `padding`, **`white-space`**를 웹/태블릿 값으로 표에 정리한다. "안 바뀔 것 같다"는 이유로 표에서 빼지 않는다 — 대조해서 "동일함"이라고 확인한 것도 표에 넣는다.
   - **`flex-direction`이 `row`→`column`으로 바뀌는 지점은 특히 주의한다.** 부모에 `white-space: nowrap`이 걸려있으면 자식에게 상속되는데, 행에서 열로 바뀐 뒤에도 이 상속을 그대로 두면 자식 텍스트가 카드 폭보다 훨씬 넓게 한 줄로 밀려나고, 부모/조상에 `overflow: hidden`이 있으면 화면 밖으로 잘려서 **텍스트가 안 보이는 사고**로 이어진다(실제 사례: 배지 이름 텍스트가 모바일에서 통째로 사라짐). `flex-direction`을 `column`으로 바꾸는 브레이크포인트에서는, 그 조상 트리에 `white-space: nowrap`이 걸려있는 게 있는지 확인하고, 세로 스택에 맞게 `white-space: normal`(필요시 `ellipsis` 유지하려면 `overflow-wrap`/`word-break`도 같이 검토)로 재설정한다.
   ```
   | 요소(클래스) | 웹 | 태블릿 |
   |---|---|---|
   | .card | width:250px, flex-row | width:100%, flex-row |
   | .card__img | height:140px | height:140px (동일 확인) |
   ```
4. 표가 다 채워지면, **태블릿 전용 `@media (max-width: 1295px)` 블록 하나**로 CSS를 작성한다. 데스크톱 고정폭이 남아있는 요소는 표의 유동값으로 오버라이드한다. 이 브레이크포인트 값 자체는 임의로 바꾸지 않는다(회사 표준, `layout.css` 주석 참고).
5. **이 단계 결과(대조 표 또는 유동 처리 내역 + 작성한 태블릿 CSS)를 보여주고, 모바일 링크를 요청한다.** 여기서 멈추고 다음 단계로 넘어가지 않는다 — 사용자가 모바일 링크를 줄 때까지 기다린다.

## 3단계 — 모바일

1. **모바일 피그마 링크가 있는지 먼저 확인한다.** `AskUserQuestion`으로 묻는다: "모바일 전용 피그마 프레임이 있나요, 아니면 없어서 웹/태블릿 기준으로 안 깨지지 않게만 처리할까요?"
   - **모바일 프레임이 있으면**: 그 링크를 받아서 아래 2~4번대로 정확히 대조해서 만든다.
   - **모바일 프레임이 없으면**: 구체적인 모바일 전용 디자인을 추측해서 만들지 않는다. 대신 웹(또는 태블릿) 레이아웃을 기준으로 "화면이 안 깨지고 자연스럽게 흐르는" 수준의 합리적인 처리만 한다(고정폭 요소 유동화, 세로 스택 전환 정도). 세부적인 재배치는 하지 않는다 — 애매하면 사용자에게 확인한다. 이 경우 4번(보고)에 "모바일 전용 프레임 없이 웹/태블릿 기준으로 유동 처리함"이라고 명시한다.
2. (모바일 프레임이 있는 경우) 모바일 프레임을 조회한다(해당 노드만). 같은 요소 목록을 기준으로 **웹/태블릿과 각각 대조**한다(2단계와 같은 방식, 같은 표에 모바일 열을 추가).
   ```
   | 요소(클래스) | 웹 | 태블릿 | 모바일 |
   |---|---|---|---|
   | .card | width:250px, flex-row | width:100%, flex-row | width:100%, flex-col |
   | .card__img | height:140px | height:140px (동일 확인) | height:120px |
   ```
3. **모바일 전용 `@media (max-width: 744px)` 블록 하나**로 CSS를 작성한다. **태블릿 블록과 절대 합치지 않는다 — 완전히 분리된 별도 블록이어야 한다.** 데스크톱/태블릿의 고정값이 남아있는 요소는 오버라이드한다.
4. `zoom`, 반응형 목적의 `transform: scale()` 같은 비표준 편법을 쓰지 않는다. `box-sizing: border-box` 전역 리셋이 `layout.css`에 있는지 확인한다(없으면 페이지 전체 가로 스크롤 위험).

## 4. 완료 후 보고 (필수, 이 형식 그대로)

- 조회한 프레임의 노드 ID (웹은 항상, 태블릿/모바일은 프레임이 있었던 경우에만 — 없었다면 각각 "프레임 없음, 웹/태블릿 기준 유동 처리"라고 명시). **각 노드 ID 옆에 "사용자가 준 링크"라고 반드시 표시한다.** 만약 어느 하나라도 사용자가 준 링크가 아니라 스스로 찾은 것이라면, 그 사실을 숨기지 말고 명시하고 즉시 사용자에게 맞는 링크인지 확인받는다.
- 최종 완성된 대조 표 전체 (웹/태블릿/모바일 3열)
- 태블릿 `@media` 블록과 모바일 `@media` 블록이 각각 **별도로** 존재하는지 재확인한 결과 (하나로 합쳐지지 않았는지)
- 페이지 전체 기준으로 의도치 않은 가로 스크롤이 생기지 않는지 확인한 결과

**이 보고에 태블릿이나 모바일 관련 내용이 하나라도 빠져 있다면, 그건 이 스킬이 제대로 실행되지 않은 것이다.**

**일반 원칙 — 요소가 화면에서 완전히 안 보이면(사라지면)**: `white-space` 상속은 그런 사고의 한 가지 원인일 뿐이다. 브레이크포인트 전환 후 어떤 요소가 통째로 안 보이는 문제를 발견하면, 개별 속성을 하나씩 의심하기 전에 먼저 **조상 트리에 `overflow: hidden`이 있는지, 그리고 문제의 요소가 실제로는 그 조상보다 훨씬 크게 렌더링되고 있어서 화면 밖으로 잘려나간 건 아닌지**부터 확인한다. `flex-shrink`, `white-space`, 고정 `width` 등 행↔열 전환 때 재검토가 필요한 속성이 남아있으면 이런 사고가 반복될 수 있다.

## 수정 요청 처리

이미 만든 태블릿/모바일 결과에 문제가 있다고 사용자가 알려주면(예: ".card__tag가 이상해요", "모바일에서 텍스트가 사라져요"), 처음부터 다시 만들지 않는다. 대신:

1. **문제가 있다고 지목된 브레이크포인트(태블릿 또는 모바일, 혹은 둘 다)의 프레임을 다시 조회한다.** 사용자가 새 링크를 안 줬으면, 그 브레이크포인트를 만들 때 이미 조회했던 노드 ID로 다시 조회한다(새로 찾지 않는다 — 어느 노드였는지는 이전 보고에 이미 남겨뒀어야 한다).
2. **재조회한 데이터와 지금 코드의 대조는 같은 세션이 직접 하지 말고 `figma-review-agent`(Opus, 독립 컨텍스트)를 Task 도구로 호출해서 맡긴다.** 방금 만든 코드를 만든 사람이 다시 봐서 "괜찮아 보이는데"로 넘기는 걸 막기 위해서다. 문제 영역과 관련된 요소들 위주로 대조하게 하고(전체 페이지를 처음부터 다시 대조하지 않는다), 검증 결과를 받는다.
3. 검증에서 나온 문제의 원인을 구체적으로 짚고(예: "부모에 `align-items`가 빠져서 자식이 늘어났다"), 그 부분만 수정한다. 문제없는 다른 부분까지 다시 만들지 않는다.
4. 수정 후, 왜 그런 문제가 있었는지와 무엇을 고쳤는지 보고한다.

사용자가 "처음부터 다시 만들어줘"라고 명확히 요청했을 때만 해당 브레이크포인트를 완전히 재생성한다.

## 방향성 있는 반복 요소(화살표, 커넥터 등)

같은 요소가 레이아웃 안에서 여러 번 반복되는데, **위치에 따라 방향/회전이 달라져야 하는 경우**를 놓치기 쉽다. 예: 그리드 레이아웃에서 같은 행 안의 카드끼리 잇는 화살표는 가로, 다음 행으로 넘어가는 화살표는 세로로 회전된 채로 나오는 경우. 이걸 "모바일에서 화살표는 다 똑같이 작아지고 숨겨지고 보이고만 하면 되겠지"로 크기/표시여부만 다루고 회전은 빠뜨리지 않는다. 참고 코드에서 같은 아이콘/화살표라도 인스턴스마다 `rotate` 값이 다르게 들어있는지 확인하고(예: `-rotate-90`), 다르면 그 자리에 맞는 회전을 CSS(`transform: rotate(...)`) 로 적용한다. 반복 요소 하나하나가 정말 다 같은 처리를 받아야 하는지, 아니면 위치별로 다른 처리(회전, 다른 이미지 에셋 등)가 필요한지 참고 코드에서 인스턴스마다 확인한다.

## CSS만으로 해결 안 되는 구조적 차이가 있을 때

기본적으로 이 스킬은 **웹 HTML 구조는 그대로 두고 CSS(미디어 쿼리)만 추가**하는 걸 원칙으로 한다. 하지만 태블릿/모바일 디자인이 웹과 근본적으로 다른 배치(단순 크기/방향 변경이 아니라, 요소 순서 자체가 크게 바뀌거나 완전히 다른 그룹핑이 필요한 경우)를 요구해서 CSS 오버라이드만으로는 구현이 안 될 수 있다. 이럴 때는 다음 순서로 처리한다.

1. **먼저 HTML을 안 건드리고 해결되는 CSS 기법부터 시도한다.**
   - 순서만 바뀌면 되는 경우: `order` 속성(flex/grid 자식에)으로 DOM 순서는 그대로 두고 시각적 순서만 바꾼다.
   - 배치 자체가 크게 바뀌는 경우: CSS Grid의 `grid-template-areas`를 브레이크포인트마다 다르게 정의해서, DOM 구조는 그대로 두고 시각적 재배치만으로 해결 가능한지 검토한다.
2. **이 방법들로도 정말 안 되면(예: 어떤 요소가 두 개의 서로 다른 부모 그룹으로 나뉘어야 하는 등, CSS 재배치의 한계를 넘어서는 경우)**, 임의로 웹 HTML 구조를 고치지 않는다. 대신 사용자에게 명확히 알린다: "이 태블릿/모바일 디자인은 지금 웹 HTML 구조로는 CSS만으로 구현할 수 없습니다 — 구체적으로 [어떤 부분이 왜 안 되는지]. HTML 구조 자체를 바꿔야 하는데, 이러면 웹 화면에도 영향이 갈 수 있어 진행 전에 확인이 필요합니다." 사용자가 동의하면, 그때 HTML 구조를 조정하되, **조정 후 웹(미디어 쿼리 없는 기본 상태)이 여전히 원래 웹 디자인과 똑같이 보이는지 반드시 재확인한다**(구조를 바꾸다가 웹 화면 자체가 틀어지는 걸 방지).

## 회귀 방지 (기존에 맞던 부분이 깨지지 않게)

웹 따로, 태블릿 따로, 모바일 따로 — 이렇게 나눠서 작업을 진행하면(정식 3단계 순차 흐름이 아니라 시간 간격을 두고 따로따로 요청받는 경우 포함), **이번에 만지는 브레이크포인트를 고치다가 이미 맞게 되어 있던 다른 브레이크포인트의 같은 클래스 규칙을 실수로 건드리는 사고**가 날 수 있다. 이를 막기 위해:

1. **어떤 클래스의 CSS를 수정하기 전에, 그 클래스가 다른 `@media` 블록(다른 브레이크포인트)에도 정의되어 있는지 먼저 확인한다.** 지금 작업 중인 블록만 보고 그 클래스를 통째로 다시 쓰지 않는다 — 다른 블록에 있는 규칙은 손대지 않고 그대로 둔다.
2. **수정 후, 이번에 손대지 않으려던 다른 브레이크포인트 블록들이 실제로 그대로인지 다시 확인한다.** 파일을 통째로 다시 훑어서, 이번 작업과 무관한 블록의 내용이 삭제되거나 값이 바뀌지 않았는지 확인한다.
3. 특히 **여러 아이콘/변형 요소마다 개별 클래스(`.icon--a`, `.icon--b` 등)로 나뉘어 있는 경우**, 이번에 관련된 것만 고치고 나머지는 그대로 둔다 — 비슷비슷한 클래스가 많으면 전체를 한꺼번에 다시 쓰다가 무관한 것까지 값이 바뀌는 사고가 나기 쉽다.
