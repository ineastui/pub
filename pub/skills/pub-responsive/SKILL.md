# 웹 페이지에 태블릿/모바일 반응형 추가

이 스킬은 **딱 하나의 일만** 한다: 이미 있는(또는 지금 만드는) 웹(데스크톱) CSS에, 태블릿과 모바일 오버라이드를 빠짐없이 추가하는 것. 그 외의 것(폼 요소 시맨틱 처리, 이미지 규칙, GNB/LNB 재사용, 클래스 네이밍 등)은 `/pub` 스킬의 몫이니 여기서 다시 다루지 않는다.

**링크는 한 번에 여러 개를 받지 않는다. 웹 → 태블릿 → 모바일 순서로 하나씩 받아서, 그 단계를 완전히 끝내고 나서 다음 링크를 요청한다.** 한꺼번에 여러 링크를 받으면 뒤쪽(특히 모바일)을 놓치는 경우가 실제로 반복됐다 — 단계를 쪼개서 한 번에 하나만 처리하는 게 훨씬 확실하다.

**"링크를 받는다"는 반드시 사용자가 직접 타이핑하거나 붙여넣은 링크만을 뜻한다.** 피그마 파일을 검색하거나 목록을 조회해서 "이게 태블릿/모바일 프레임인 것 같다"고 스스로 찾아서 쓰지 않는다 — 이름이 같은 프레임이 여러 개 있는 경우가 많아서 자동으로 찾으면 잘못된 프레임을 쓸 위험이 크다. 사용자가 링크를 주기 전까지는 어떤 노드도 조회하지 않는다. 사용자가 링크 없이 "있다"/"없다"로만 답하면, 있다고 했는데 링크를 안 줬다면 다시 링크를 요청한다 — 알아서 찾지 않는다.

## 1단계 — 웹

1. **대상 CSS 파일 경로를 확인한다** (예: `inc/{영역}/css/design.css`).
2. **웹 피그마 링크를 하나 받는다.** 아직 없으면 이것부터 물어보되, 처음 요청할 때 전체 흐름을 먼저 짧게 안내한다: "반응형은 웹 → 태블릿 → 모바일 순서로 하나씩 진행할게요. 먼저 웹 프레임 링크를 알려주세요." 이렇게 처음부터 전체 순서를 알려줘야, 왜 지금 링크 하나만 요청하는지 사용자가 헷갈리지 않는다(이 스킬을 처음 쓰는 사람도 있으므로).
3. 이 링크로 가리키는 컴포넌트가 대상 CSS 파일에 이미 만들어져 있는지 확인한다. 없으면(웹 버전 자체가 아직 없으면) `/pub`으로 웹 버전부터 만들어달라고 안내하고 여기서 중단한다.
4. 웹 프레임을 Figma 도구로 조회한다(해당 노드만, 페이지 전체 조회 금지). 참고 코드를 최상위부터 리프 요소까지 훑어서, 이번에 반응형을 적용할 요소들의 목록을 정리한다(이게 다음 단계 대조의 기준이 된다).
5. **이 단계가 끝났다고 사용자에게 알리고, 태블릿 링크를 요청한다.** 웹 단계 결과를 먼저 간단히 보여준다(예: "웹 프레임 분석 완료 — .card, .card__img, .card__body 등 N개 요소 확인. 이제 태블릿 링크를 알려주세요.")

## 2단계 — 태블릿

1. **태블릿 피그마 링크가 있는지 먼저 확인한다.** `AskUserQuestion`으로 묻는다: "태블릿 전용 피그마 프레임이 있나요, 아니면 없어서 웹 기준으로 안 깨지지 않게만 처리할까요?"
   - **태블릿 프레임이 있으면**: 그 링크를 받아서 아래 2~4번대로 정확히 대조해서 만든다.
   - **태블릿 프레임이 없으면**: 구체적인 태블릿 전용 디자인을 추측해서 만들지 않는다. 대신 웹 레이아웃을 기준으로 "화면이 안 깨지고 자연스럽게 흐르는" 수준의 합리적인 처리만 한다(예: 고정폭 요소를 `width:100%`/`flex-wrap`으로 유동화, 지나치게 넓은 gap/padding을 살짝 줄이는 정도). 세부적인 재배치(요소 순서 변경, 새로운 레이아웃 구조)는 하지 않는다 — 애매하면 규칙에 따라 사용자에게 확인한다. 이 경우 4번(보고)에 "태블릿 전용 프레임 없이 웹 기준으로 유동 처리함"이라고 명시한다.
2. (태블릿 프레임이 있는 경우) 태블릿 프레임을 조회한다(해당 노드만). 1단계에서 정리한 요소 목록을 기준으로, **하나하나 웹과 태블릿을 대조**한다 — 부모 컨테이너만 보고 끝내지 않는다. 부모의 배치(행↔열)가 바뀌었다고 자식(카드 안 텍스트, 아이콘, 배지 등)도 자동으로 확인됐다고 넘겨짚지 않는다.
3. 각 요소마다 `display`, `flex-direction`, `align-items`, `justify-content`, `gap`, `width`, `height`, `padding`을 웹/태블릿 값으로 표에 정리한다. "안 바뀔 것 같다"는 이유로 표에서 빼지 않는다 — 대조해서 "동일함"이라고 확인한 것도 표에 넣는다.
   ```
   | 요소(클래스) | 웹 | 태블릿 |
   |---|---|---|
   | .card | width:250px, flex-row | width:100%, flex-row |
   | .card__img | height:140px | height:140px (동일 확인) |
   ```
4. 표가 다 채워지면, **태블릿 전용 `@media (max-width: 1295px)` 블록 하나**로 CSS를 작성한다. 데스크톱 고정폭이 남아있는 요소는 표의 유동값으로 오버라이드한다. 이 브레이크포인트 값 자체는 임의로 바꾸지 않는다(회사 표준, `layout.css` 주석 참고).
5. **이 단계 결과(대조 표 또는 유동 처리 내역 + 작성한 태블릿 CSS)를 보여주고, 모바일 링크를 요청한다.** 여기서 멈추고 다음 단계로 넘어가지 않는다 — 사용자가 모바일 링크를 줄 때까지 기다린다.

## 3단계 — 모바일

1. **모바일 피그마 링크가 있는지 먼저 확인한다.** `AskUserQuestion`으로 묻는다: "모바일 전용 피그마 프레임이 있나요, 아니면 없어서 웹/태블릿 기준으로 안 깨지지 않게만 처리할까요?"
   - **모바일 프레임이 있으면**: 그 링크를 받아서 아래 2~4번대로 정확히 대조해서 만든다.
   - **모바일 프레임이 없으면**: 구체적인 모바일 전용 디자인을 추측해서 만들지 않는다. 대신 웹(또는 태블릿) 레이아웃을 기준으로 "화면이 안 깨지고 자연스럽게 흐르는" 수준의 합리적인 처리만 한다(고정폭 요소 유동화, 세로 스택 전환 정도). 세부적인 재배치는 하지 않는다 — 애매하면 사용자에게 확인한다. 이 경우 4번(보고)에 "모바일 전용 프레임 없이 웹/태블릿 기준으로 유동 처리함"이라고 명시한다.
2. (모바일 프레임이 있는 경우) 모바일 프레임을 조회한다(해당 노드만). 같은 요소 목록을 기준으로 **웹/태블릿과 각각 대조**한다(2단계와 같은 방식, 같은 표에 모바일 열을 추가).
   ```
   | 요소(클래스) | 웹 | 태블릿 | 모바일 |
   |---|---|---|---|
   | .card | width:250px, flex-row | width:100%, flex-row | width:100%, flex-col |
   | .card__img | height:140px | height:140px (동일 확인) | height:120px |
   ```
3. **모바일 전용 `@media (max-width: 744px)` 블록 하나**로 CSS를 작성한다. **태블릿 블록과 절대 합치지 않는다 — 완전히 분리된 별도 블록이어야 한다.** 데스크톱/태블릿의 고정값이 남아있는 요소는 오버라이드한다.
4. `zoom`, 반응형 목적의 `transform: scale()` 같은 비표준 편법을 쓰지 않는다. `box-sizing: border-box` 전역 리셋이 `layout.css`에 있는지 확인한다(없으면 페이지 전체 가로 스크롤 위험).

## 4. 완료 후 보고 (필수, 이 형식 그대로)

- 조회한 프레임의 노드 ID (웹은 항상, 태블릿/모바일은 프레임이 있었던 경우에만 — 없었다면 각각 "프레임 없음, 웹/태블릿 기준 유동 처리"라고 명시). **각 노드 ID 옆에 "사용자가 준 링크"라고 반드시 표시한다.** 만약 어느 하나라도 사용자가 준 링크가 아니라 스스로 찾은 것이라면, 그 사실을 숨기지 말고 명시하고 즉시 사용자에게 맞는 링크인지 확인받는다.
- 최종 완성된 대조 표 전체 (웹/태블릿/모바일 3열)
- 태블릿 `@media` 블록과 모바일 `@media` 블록이 각각 **별도로** 존재하는지 재확인한 결과 (하나로 합쳐지지 않았는지)
- 페이지 전체 기준으로 의도치 않은 가로 스크롤이 생기지 않는지 확인한 결과

**이 보고에 태블릿이나 모바일 관련 내용이 하나라도 빠져 있다면, 그건 이 스킬이 제대로 실행되지 않은 것이다.**

## 다음 진행 확인

작업을 마치면 `AskUserQuestion`으로 다음 중 뭘 원하는지 물어본다: 다른 컴포넌트에도 반응형 추가 / 지금 결과 수정 / 여기서 마무리. 답을 받을 때까지 다음으로 넘어가지 않는다.
